# Mod Editor ⇄ default-project Convention Unification — Plan (Option A)

**Status:** Plan
**Date:** 2026-06-30
**Repo:** `tsic-definition-editor` (this repo)
**Goal:** Make the editor read and write the **`default-project` repo convention natively**, so it can author into and (later) push to `Chico-Games/tsic-default-project` — the single source of truth that TSIC and the LevelEditor already consume.

## Why Option A

The per-definition files are **already identical** (typed-envelope/lean, `{id, asset_path, class, schema_version, properties}`; lean⇄envelope handled at the DataSource boundary via the pack's `_schema.json` in `dataSource.ts`). Only the **control/meta layer** differs. Option A teaches the editor that meta layer directly — no translation/adapter shim (matches the project's "no transform layers" preference). Option B (an adapter) was rejected: it still has to solve `mod.json` + sidecar handling and adds a layer to maintain.

## Current state — the three mismatches (verified 2026-06-30)

| Concern | Editor today | default-project | 
|---|---|---|
| Manifest | `manifest.json` — `{ folders:[], files:[{folder,ids}] }` | `.manifest.json` — `{ schema_version, generated_at, assets:{ <folder>:{ <id>:<UE-asset-path> } } }` |
| Identity / version | `default.json` — `{ schema_version, version, label, published_at }` (publish counter) | `mod.json` — `{ id, displayName, version }` (mod **identity**, what TSIC `ScpModDiscovery` + `sync-biomes.mjs` key on) |
| Schema sidecars | App-global, bundled at `web/public/schema/` (undotted: `class-hierarchy.json`, `property-meta.json`, `gameplay-tags.json`), loaded once by `appSchemaStore` (`appSchemaStore.ts:182`) | Per-mod dotted root sidecars: `.class-hierarchy.json`, `.property-meta.json`, `.gameplay-tags.json`, `_schema.json`, `.asset-refs.json` (UE-generated by `run_export.py`) |

**Key consequence:** the editor never writes `mod.json`, so a folder it "publishes" is invisible to the game/map-editor; and its publisher writes `manifest.json`/`default.json` (wrong names/shape) and would drop the dotted sidecars.

**Already compatible (do NOT touch):** per-record file format; lean⇄envelope conversion via per-pack `_schema.json` (`dataSource.ts` — `FsaDataSource` lazy-loads the *target's* `_schema.json`, **verified**, so publish round-trip fidelity does not depend on the bundled schema — that's only for property-grid rendering/drift).

> **AUDIT FINDING (2026-06-30) — `.manifest.json` is an asset INDEX, not a file list.** On the current `Mods/com.chicogames.default`, `.manifest.json assets` lists **37** folders but **39** definition folders exist on disk. The missing ones — `hotkey_definitions`, `input_behavior_definitions`, `situation_definitions` — are **data-only defs** (no UE `.uasset`, hence no `asset_path`, hence excluded from the asset index). **131 / 2328 records have no `asset_path`.** Consequence: you **cannot** enumerate the work-list from `.manifest.json assets` (it silently drops whole categories) and you **cannot** rebuild a complete file list from `asset_path`. Enumerate by **directory scan** instead (see Phase 1). The editor's own `manifest.json` (and the test `loadAllDefinitions`) already enumerate completely; only default-project's `.manifest.json` is the incomplete asset index.

## Design decisions (settle before coding)

1. **Dual-convention detection, not replacement.** The editor must still load its own bundled `starter-project` (editor convention) *and* a `default-project` checkout (dotted convention). Detect the target by probing for `mod.json` / `.manifest.json` (dotted = default-project) vs `manifest.json` (editor). **But do NOT use either manifest as the file enumeration for default-project** — directory-scan instead (decision 5).
2. **`mod.json.version` is a semver string — do NOT fold the publish counter into it.** *(Revised after audit.)* `mod.json` = `{id, displayName, version:"0.1.0"}` is the mod's **release identity** (semver string, human-set). The editor's `default.json.version` is an auto-incremented **integer publish counter** — a different concept and type. Keep them separate: **preserve `mod.json` untouched** as identity, and keep the editor's publish counter in `default.json` *alongside* it (or omit the counter for default-project targets). Never `+1` the semver. Surfacing a "bump release version" affordance (semver) is a separate UX feature, out of scope.
3. **Sidecars are passthrough; manifest is regenerated as an asset-index only.** *(Revised after audit.)* The UE-derived dotted sidecars (`.class-hierarchy.json`, `.property-meta.json`, `_schema.json`, `.asset-refs.json`, `.gameplay-tags.json`) are **preserved untouched** — the editor authors data, not schema. `.manifest.json` is regenerated as the **asset index**: `assets:{folder:{id:asset_path}}` built **only from records that have a non-empty `asset_path`** (data-only defs are intentionally excluded, matching the real format). (The existing publisher delete-loop iterates `folder/id` record keys only, so root sidecars are safe from deletion — keep it that way.)
4. **Project-scoped schema is a later phase.** Round-trip correctness uses per-pack `_schema.json` (already wired, verified). Making `appSchemaStore` prefer the open mod's dotted sidecars over the bundled ones is a rendering-fidelity improvement, deferrable to Phase 3.
5. **Enumerate definitions by directory scan, not by manifest — and make the scan folder-name-agnostic.** *(New, from audit.)* For a default-project (FSA) target, walk every non-dotfile subdirectory and load every `*.json`, **keeping any file with a top-level string `id`** (skip sidecars/catalogs, which lack `id`). Do **not** hardcode a folder allowlist: the test's `loadAllDefinitions` rule (`*_definitions` + `furniture_upgrade_recipe` + `layout_definitions`) is **incomplete for production** — it omits `scp_game_data`, which the editor's `manifest.json` includes and the real default-project ships. A shape-based filter (`id` present) covers `_definitions`, `furniture_upgrade_recipe`, `scp_game_data`, the data-only categories (`hotkey_/input_behavior_/situation_definitions`), and any future folder, while skipping `.assets/`, dotted sidecars, and `_schema.json`. The HTTP loader (bundled starter-project) keeps using its complete `manifest.json` since HTTP has no directory listing.

## Phases

### Phase 1 — Loader reads default-project convention
`defaultProject.ts` → `loadDefaultProjectFromFsa` (the user-opened-folder path; HTTP/bundled path is unchanged):
- **Convention detection:** if the folder has `.manifest.json` (or `mod.json`), treat it as default-project. ⚠️ The current code (`defaultProject.ts:107`) does an **unconditional** `handle.getFileHandle('manifest.json')` that *throws* on a default-project — replace it with the detection so the dotted path doesn't ENOENT before it starts.
- **Enumerate by directory scan** (NOT the manifest): iterate the folder's directory handles, skip dotfile dirs (`.assets` etc.), read every `*.json`, and keep those with a top-level string `id` (shape filter — see decision 5). This loads the data-only categories `.manifest.json` omits *and* `scp_game_data`, without a brittle folder allowlist. (Editor-convention folders keep using `manifest.json`.)
- **Identity:** read `mod.json` when present → carry `{id, displayName, version}` as identity (do **not** coerce the semver into the integer counter). Read `default.json` for the integer publish counter if present; else `FALLBACK_META`. Tolerate both missing.
- Tests: extend `defaultProject.test.ts` / `definitionsStoreLoadDefault.test.ts` with a dotted fixture that **includes a data-only def folder** (e.g. `situation_definitions`) and assert it loads. Add a real-`default-project` test gated on availability like `packDir.ts`.

### Phase 2 — Publisher writes default-project convention
`defaultPublisher.ts` → `publishAsNewDefaultVersion`:
- Detect target convention (presence of `.manifest.json`/`mod.json`). ⚠️ The current entry guard (`defaultPublisher.ts:38`) is `if (!hasFile(target, 'manifest.json')) throw …` — it **rejects a real default-project folder**. Widen it to accept `.manifest.json`/`mod.json` (else publish-into-default-project throws before writing anything).
- **Write `.manifest.json` as an asset index:** `{ schema_version, generated_at, assets:{folder:{id:asset_path}} }`, including **only records with a non-empty `asset_path`** (skip data-only defs). Match the real key order/shape.
- **Preserve `mod.json` untouched** (identity). Keep the integer publish counter in `default.json` (or omit for dotted targets) — never bump the semver.
- Explicitly do **not** write or delete the dotted sidecars (passthrough). Keep the delete-loop scoped to `folder/id` record keys.
- Keep the legacy `manifest.json`/`default.json` path for editor-convention targets.
- Tests: extend `defaultPublisher.test.ts` — publish into a dotted target containing a data-only def; assert `.manifest.json` excludes the data-only entries, `mod.json` is byte-unchanged, dotted sidecars untouched, record bytes lean-identical, and the data-only def files themselves are still written/kept.

### Phase 3 — (Optional) project-scoped schema
`appSchemaStore.ts`: when a `default-project` is open, prefer its dotted `.class-hierarchy.json`/`.property-meta.json` over the bundled `web/public/schema/` copies; fall back to bundled. Improves property-grid/drift fidelity for mods whose schema differs from the bundled snapshot. Update `appSchemaStore.test.ts`, `schemaDriftValidator.test.ts`.

### Phase 4 — Wire the git push (separate effort)
Once round-trip is proven: add a `default-project` remote to this repo, vendor it as a subtree (or point "Open project" at a checkout), and add a publish→`git subtree push … beta` flow. Out of scope for this plan; tracked in the TSIC consumer-wiring plan.

## Risks / limitations

- **Sidecar staleness:** editor edits that add/remove defs won't refresh the UE-derived `.asset-refs.json`/`.gameplay-tags.json`/`.class-hierarchy.json`; only `.manifest.json` is regenerated. A drift check (TSIC `AggregateSha1Hex`) may warn until the next UE re-export. Acceptable — the editor is a data tool, not a UE exporter. Document it.
- **Data-only defs are first-class but invisible to `.manifest.json`:** `hotkey_/input_behavior_/situation_definitions` (and any def with no `asset_path`) must load (via directory scan) and save (their `*.json` written) but are intentionally **absent** from the regenerated `.manifest.json assets`. Tests must cover that they survive a load→publish round-trip even though the manifest never lists them.
- **New asset-backed records:** a brand-new asset-backed def authored in the editor needs an `asset_path` to appear in `.manifest.json`; if blank it's silently treated as data-only. Decide a convention (derive from id/folder, or require the field) if authoring brand-new assets is in scope.
- **Two conventions in the loader** add branches; keep them isolated behind a single `detectConvention()` + a shared directory-scan enumerator so the rest of the loader is shape-agnostic.
- **`mod.json` semver is not the editor's counter** — keep the two version concepts separate; don't display/auto-edit the semver as if it were the publish counter.

## Acceptance

- Load a real `Chico-Games/tsic-default-project` checkout (dotted `.manifest.json` + `mod.json`) → **all** records render, **including** `hotkey_/input_behavior_/situation_definitions` (the categories `.manifest.json` omits).
- Edit a value, publish → target's record bytes change lean-identically; `.manifest.json` regenerated as an asset-index (data-only defs excluded); **`mod.json` byte-unchanged**; dotted sidecars unchanged; data-only def files still present.
- Bundled `starter-project` still loads/publishes (no regression).
- `npm run typecheck` + `npm run test` green.
